# 开发约束

## 总原则

- 不要改用户没有要求的功能和视觉区域。
- 不要为了局部效果复制一套新的布局系统。
- 修改前先读现有模块和测试，优先沿用已有模式。
- 用户关心的是实机/截图里的体验是否顺，不是代码看起来理论正确。
- 每次宣称完成前必须有新鲜验证结果。

## UI 约束

- 顶部固定栏高度要全局统一，不能页面各自写一套高度。
- 返回键位置要全局统一，以歌手详情页已经认可的位置为基准。
- 搜索框只在点击搜索按钮后展开；再次点击搜索按钮，或空输入时点击外部，应自动关闭。
- 搜索框不能压到搜索按钮和设置按钮，按钮尺寸要一致。
- 歌单详情、搜索结果、歌手详情、专辑详情的顶部间距要协调。
- 歌曲列表密度要保持一致，尤其是歌单详情、搜索结果、歌手详情、专辑详情。
- 播放详情页播放队列应是顺滑的上抽屉，不要出现卡顿的下滑过程。
- 歌词滑动参考首页歌词，优先整体滑动，不做生硬跳动。
- 点击态只用于反馈，不要误做成持续选中态，除非需求明确要求持续选中。
- 按钮点击后不要出现浏览器默认蓝色焦点框。

## CSS 约束

核心 CSS 方法和关键区域要保留中文注释，尤其是：

- 全局顶部控制区。
- 返回键/搜索框/设置按钮位置。
- 歌曲列表密度和行状态。
- 播放队列抽屉。
- 歌词滚动/选中/播放按钮。

新增 CSS 时优先：

1. 找已有变量或同类选择器。
2. 把通用规则放到统一位置。
3. 页面差异只写最小必要覆盖。
4. 避免 `!important`，除非是在处理历史遗留且有注释说明。
5. 改完跑 `npm.cmd run check:css`。

不要新增无意义的单页 class 来解决全局问题。例如返回键、顶部栏、歌曲行这些都应统一。

## 播放与喜欢逻辑约束

- 喜欢按钮的图标和逻辑要与首页播放器一致。
- 重启应用后，喜欢状态要从歌单、远端或本地状态正确刷新。
- 重启应用后不应恢复到上次播放秒数，当前要求是从 `0` 开始。
- 歌词快进必须是先选中歌词，再点击播放按钮。未选中时点到按钮区域不能快进。
- 播放队列序号显示为 `1, 2, 3`，不要补零。

## 后端与数据约束

- `.env` 不提交。
- `data/` 不提交。
- SQLite、缓存、用户背景、音频缓存都属于运行时数据。
- 第三方 API key 不写进代码，不写进公开文档。
- 没有 API key 时应用要尽量 fallback，而不是直接崩。

## Android 约束

- `android/local.properties` 只保留本机路径，不提交。
- 不提交 Android build 输出、`.cxx/`、`.gradle/`、`nodejs-project/`、`cdvnodejsmobile/` 等生成物。
- 打包前确保执行 Android 准备脚本，不能只依赖 Android Studio 自动构建。
- Android 打包出问题时优先检查 npm disturl、SDK/NDK 路径、nodejs-mobile-cordova patch。

## Git 约束

不要提交：

- `.env`
- `node_modules/`
- `data/`
- `android/local.properties`
- `android/**/build/`
- `android/**/.cxx/`
- APK/AAR 等构建产物
- 本地截图、日志、临时文件

提交前建议：

```powershell
git status --short
npm.cmd run check
npm.cmd run check:css
```

如果只是本地开发文档，可以先不推 GitHub；如果要公开，先确认没有本机路径、账号、密钥、私密截图。
